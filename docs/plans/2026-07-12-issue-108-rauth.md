# Issue #108 RAUTH Implementation Plan

## Architecture boundary

- Capability owner: `platform_foundation.auth_platform`.
- Protected surfaces: every AI-CRM owned admin, internal bearer, scoped bearer,
  worker/agent and partner-webhook route. Supplier OAuth/access tokens and
  supplier callback signatures remain owned by `integration_gateway`.
- Real external calls: none during implementation or reconciliation. The
  production release may only perform the approved upstream WeCom login flow;
  existing business effect gates remain unchanged.
- Production data: auth principals, clients, keys, sessions, grants, tokens,
  revocations, replay nonces and security audit metadata. Secrets are hashed or
  envelope-encrypted and never emitted by diagnostics.
- Rollback: restore the previous exact release and runtime configuration as one
  unit. There is no legacy credential fallback or dual-stack runtime.

## Delivery sequence (one branch, one PR, one release)

1. Freeze the credential surface
   - Generate a checked-in inventory of route auth schemes, direct credential
     reads/comparisons, service callers, webhooks and runtime secret keys.
   - Add a static gate that fails on new route-local token comparisons.

2. Select and document the standards implementation
   - Add an ADR for the OAuth 2.0/OIDC implementation and pinned dependencies.
   - Define the threat model, opaque-token model, sender constraints, key
     lifecycle, access/refresh TTLs and whole-release rollback.

3. Build the Auth Platform
   - Add migration 0104 for principals, clients, keys, grants, sessions,
     authorization codes, access/refresh tokens, revocations, replay nonces and
     security audit events.
   - Store client secrets, access tokens, refresh tokens and authorization codes
     as keyed hashes; encrypt private key material with the existing secret-store
     boundary when server-held keys are unavoidable.
   - Implement repository, application services, policy evaluator and immutable
     `AuthContext`.

4. Implement standards endpoints and middleware
   - Discovery, authorization, token, introspection, revocation, JWKS/userinfo
     where applicable, and server-side BFF session endpoints.
   - Authorization Code + PKCE, Client Credentials + `private_key_jwt`, refresh
     rotation/family revocation and delegated token exchange.
   - mTLS/DPoP proof binding and HMAC/HTTP-signature partner webhook replay
     protection.

5. Cut every caller to AuthContext
   - Replace admin signed cookie authorization with the BFF session backed by
     Auth Platform state.
   - Replace `internal_bearer` and `scoped_bearer` shared environment tokens with
     opaque access-token introspection and route-policy scope/resource checks.
   - Give each worker/timer/diagnostic and external agent its own service
     principal, client, audience and least-privilege scope.
   - Adapt AI-CRM partner webhooks to signed requests while preserving supplier
     callback protocols unchanged.

6. Delete the old implementation
   - Remove shared-token lookup/comparison code, direct route token readers,
     signed admin-session authorization, legacy fallback flags and retired secret
     environment declarations.
   - Regenerate route, runtime, high-risk, ownership and lifecycle inventories.

7. Verify and release
   - PostgreSQL migration/concurrency/fault tests; OAuth/OIDC protocol, token
     rotation/reuse, revocation, audience/scope/resource, sender-constraint,
     replay and log-redaction tests.
   - Full architecture, backend, frontend and dependency gates.
   - Bootstrap all required production principals/clients/keys before switching;
     deploy one exact SHA, restart all runtime units, and run count-only auth
     reconciliation with zero legacy credential use and zero unmigrated caller.

## Non-goals

- No new business feature, page, metric or context.
- No replacement of WeCom/WeChat Pay supplier protocols.
- No weakening of approval, CSRF/step-up, idempotency, rate-limit or external
  effect safety controls.
- No shadow auth, compatibility fallback or deferred caller migration.
